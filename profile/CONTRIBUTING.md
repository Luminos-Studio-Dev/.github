# 🤝 貢献ガイドライン (CONTRIBUTING) - Luminos-Studio-Dev

Luminos-Studio-Dev のプロジェクトに興味を持っていただき、誠にありがとうございます！
私たちの開発はコミュニティからの貢献によって成り立っています。このガイドラインを読み、円滑な協力体制を築きましょう。

## ✨ 私たちのビジョンと行動規範

Luminos-Studio-Dev は、すべての貢献者が安心して参加できる、**創造的で理想的な開発環境**を目指しています。

* **行動規範 (Code of Conduct):**
    すべての参加者は、[CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md) に従う必要があります。敬意を持った建設的な議論を心がけてください。

## 🚀 貢献の始め方

貢献の形式はコードの修正に限りません。以下のいずれの方法でも歓迎いたします。

### 1. バグ報告 (Bug Reports)

予期せぬ動作やエラーを発見した場合、Issueを作成してください。

* **Issueテンプレートの利用:** 必ず「**バグ報告テンプレート**」を使用し、以下の情報を含めてください。
    * 発生した環境（OS、ブラウザ、使用バージョンなど）
    * **再現手順**：どのような操作でバグが発生するかを具体的に記述してください。
    * 期待される結果と実際の結果。

### 2. 機能提案 (Feature Suggestions)

新しい機能や改善のアイデアはいつでも歓迎します。

* **Issueテンプレートの利用:** 必ず「**機能要望テンプレート**」を使用してください。
* **目的を明確に:** 「何が欲しいか」だけでなく、「**その機能が解決する問題**」や「**ユーザーにとってのメリット**」を記述してください。

### 3. コードの貢献 (Code Contribution)

コードを修正・追加する場合は、以下のワークフローに従ってください。

#### A. 開発前の確認

1.  **Issueを作成する:** 既存のバグ修正や機能追加であっても、作業開始前に必ず Issue を作成し、オーナーまたはコアメンバーの承認を得てください。
2.  **自己アサイン:** Issueに「この作業は自分が担当する」旨のコメントをし、他のメンバーとの重複を防いでください。

#### B. 開発と Pull Request (PR) 作成

1.  **リポジトリをフォーク:** 自身のGitHubアカウントにリポジトリをフォークします。
2.  **ブランチの作成:** 以下のルールでブランチを作成してください。
    * **命名規則:** `[ブランチタイプ]/[関連Issue番号]-[簡潔な説明]`
        * 例: `feat/123-add-dark-mode` (機能追加)
        * 例: `fix/456-fix-login-error` (バグ修正)
3.  **コミットメッセージの書式:**
    全てのコミットは、以下の[Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/) に準拠したプレフィックスを付けてください。
    * `feat`: 新機能の追加
    * `fix`: バグ修正
    * `docs`: ドキュメントのみの変更（READMEなど）
    * `chore`: ビルドプロセスや補助ツールの変更
    * `refactor`: リファクタリング（機能変更なし）

4.  **Pull Request (PR) を作成:**
    * **PRテンプレートの利用:** 必ず「**Pull Requestテンプレート**」を使用して、変更点を明確にしてください。
    * **参照:** PRの概要に、関連するIssue番号（例: `Closes #123`）を必ず含めてください。
    * **テスト:** 変更が意図通りに機能することを**必ずローカルでテスト**してください。

## 🚨 コードレビューとマージ

* コアメンバーがコードレビューを行い、フィードバックを行います。
* レビューコメントには迅速かつ丁寧に対応してください。
* 承認（Approve）が得られた後、コアメンバーによって `main` ブランチにマージされます。

皆様の創造的な貢献をお待ちしています！
